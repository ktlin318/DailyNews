# AI 每日新聞與話術 Agent：MVP 執行規畫

## 1. MVP 的邊界

第一版只解決一件事：每天由具備搜尋與網頁閱讀工具的 AI Agent，從有限、允許存取的新聞來源中，產出一份「每句可回指原文」的重點摘要頁。客群話術先用規則與範本生成，人工確認後使用；不在第一版自動發送 LINE、Email 或 CRM 訊息。

成功條件：AI 每日自動完成搜尋、閱讀與整理；相同事件能合併；數字、日期、時間與專有名詞能逐一驗證；每則摘要至少附一個可開啟的來源；錯誤可被標示並停止發布；網頁可依主題與客群篩選。

## 2. 建議技術架構

- 新聞探索層：具備 Web Search／Browser 工具的 AI Agent；只搜尋來源白名單、指定主題與指定時間範圍。
- 網頁閱讀層：AI 直接開啟原始新聞頁，讀取標題、正文、發布／更新時間及 canonical URL；不使用 Python 爬蟲。
- 排程層：由 Agent 平台的排程／Automation 每日觸發工作，不使用 Python 排程器。
- 儲存層：MVP 可使用網站平台資料庫、雲端資料表或 JSON 資料服務。保存來源 URL、查閱時間、引用片段與內容指紋；是否保存全文須依來源授權決定。
- 摘要層：模型只能根據本次已開啟頁面的內容輸出固定 JSON；禁止使用模型記憶補充新聞事實。
- 驗證層：使用第二次、獨立的 AI 查核工作重新開啟來源，抽取並逐項比對數字、幣別、百分比、日期、時間、人名與機構名。高風險欄位採字串完全一致或正規化後一致，不接受語意猜測。
- Web 層：目前 demo 為 React/Next 介面；後續由同一網站的 Server Route／Server Action 提供資料，不需要 Python 或 FastAPI。
- 監控層：structlog/Sentry；紀錄來源失敗率、摘要拒絕率、數字驗證失敗與人工修訂。

## 3. 資料流程與品質閘門

1. `Source Registry`：為每個網站記錄網域、允許範圍、搜尋方式、時區、語言、robots.txt 與使用條款確認日。
2. `Discovery Agent`：以白名單網域、日期與主題執行搜尋，取得候選 URL；找不到原始頁面時不得只引用搜尋結果摘要。
3. `Reading Agent`：逐一開啟新聞頁，取得標題、作者、發布／更新時間、正文證據與 canonical URL；遇到付費牆或讀取不完整時標示不可用，不繞過限制。
4. `Event Grouper`：以 canonical URL、標題、事件實體與時間做去重分群；保留所有來源，不把多篇文章的不同說法混成一個事實。
5. `Fact Extractor`：由 AI 先抽取所有不可生成欄位：數字、日期、時間、金額、比例、單位、專名及其逐字證據片段。
6. `Summarizer`：模型輸出 `summary_sentences[]`，每句包含 `evidence_spans[]` 與 `source_ids[]`；沒有證據就不得輸出。
7. `Verification Agent`：以獨立上下文重新開啟來源；所有數字 token 必須完全存在於引用證據，日期正規化後比對，專名必須出現在來源。跨來源衝突標記 `conflict`，不可自行裁決。
8. `Editorial Gate`：低於門檻、只有單一弱來源、來源衝突或解析異常者進人工佇列；通過者才寫入 published digest。
9. `Web Publisher`：前台顯示摘要、影響、信心等級、更新時間與折疊式多來源；所有連結以新分頁開啟。

## 4. AI 輸出契約（必要欄位）

每個事件包含：`event_id`、`headline`、`category`、`published_at`、`summary_sentences`、`facts`、`impact`、`source_ids`、`conflicts`、`verification_status`、`review_status`。每個摘要句必須帶 `source_id`、原文起訖位置或精確證據片段。模型溫度設低，JSON Schema 強制驗證；但低溫與 Schema 都不能取代事實比對。

## 5. 多來源呈現規則

- 卡片預設只顯示「來源名稱 × N」，避免版面雜亂。
- 展開後顯示編號來源 chip；第一來源選原始發布者或官方來源，其餘按發布時間與可信度排列。
- 同事件來源超過 4 個時只顯示前三個與「其餘 N 個」。
- 若來源說法不同，明確顯示「來源有差異」，把各自說法分開列出，不產生單一確定結論。

## 6. 工作拆解與預估順序

### Sprint 0（1–2 天）：規格與合規

- 決定首批 3–5 個來源、10–15 個主題關鍵字、更新頻率與時區。
- 逐站確認 robots.txt、著作權、付費牆與可否儲存全文；若不可存全文，只存必要欄位與引用片段。
- 定義可信來源分級、新聞重要性評分、人工審核責任與刪除政策。
- 完成 30 篇新聞的人工標準答案集，供後續驗證。

### Sprint 1（3–5 天）：AI 搜尋、閱讀與資料保存

- 建立來源白名單與每站搜尋提示範本，限制日期、主題、語言與最多結果數。
- 建立 article/source/agent_run 資料結構與唯一鍵；加入重試、節流、逾時與來源不可讀狀態。
- 產出每日 Agent 執行報告：成功開啟、重複、付費牆、內容不足、連結失效。

### Sprint 2（3–5 天）：事件分群、摘要與雙 Agent 驗證

- 實作事件分群與代表標題選擇。
- 建立 JSON Schema prompt 與摘要服務。
- 實作獨立 Verification Agent；任何無逐字證據的新增數字、日期或時間直接拒絕。
- 以標準答案集測 precision、recall、數字正確率與 evidence coverage；數字正確率目標 100%。

### Sprint 3（3–4 天）：網站資料接口與正式前台

- 網站 Server Route 提供查詢、篩選、來源展開、人工核准／退回。
- 將 demo mock data 換成 Agent 產出的結構化資料；加入 loading、empty、error、stale 狀態。
- 完成手機版、鍵盤操作與連結安全屬性。

### Sprint 4（2–3 天）：話術與營運

- 客群規則放在可版本化設定檔；話術只使用已驗證 facts。
- 提供電話、LINE、面談三種長度，但先只「複製」，不自動傳送。
- 建立每日排程、告警、人工審核畫面與回滾機制。

## 7. 必備測試

- Browser reading fixture：固定測試頁讀取結果有變化時可立刻發現。
- Unit tests：數字格式（千分位、百分比、負號）、民國/西元、跨時區、相對日期。
- Contract tests：模型輸出不得多欄、漏欄或無證據。
- Adversarial tests：原文含提示注入、表格、修正文、相互矛盾來源。
- End-to-end：由 AI 開啟一篇固定測試文章、完成摘要並送到前台，驗證來源連結與所有 fact token。
- 人工抽查：MVP 每日 100%；穩定後依風險分層抽樣，財經數字類維持高比例人工覆核。

## 8. 上線前決策清單

需要產品負責人確認：首批網站與授權、每天發布時間、新聞重要性定義、是否允許只有一個來源、客群資料是否含個資、人工核准人員、模型與雲端預算、錯誤更正與留存期限。
